## Purpose and Scope

This policy defines how {{ org() }} creates, reviews, approves, publishes, and retires information security policies. It establishes a consistent lifecycle that ensures policies are current, accessible, version-controlled, and backed by an auditable approval history.

This policy applies to all information security and compliance policies maintained in {{ org() }}'s policy repository, and covers all personnel responsible for authoring, reviewing, approving, or maintaining those policies.

## Policy Repository

{{ org() }}'s policies are stored as Markdown files in a dedicated Git repository, under `content/policies/`. The repository is the single source of truth - no policy is considered current unless it has been merged to the main branch through the process described in this policy.

On every push to the main branch, an automated pipeline compiles all policies into PDFs, builds the policy website, and deploys it. The deployment target (GitHub Pages, Azure Static Web Apps, Cloudflare Pages, or self-hosted) is configured in the repository's CI/CD workflow.

## Policy Lifecycle

### 1. Drafting

New policies must be authored on a branch, not directly on main. The branch name should describe the policy and intent, for example:

```
policy/acceptable-use-policy-initial
policy/access-control-annual-review-2025
policy/incident-response-add-ransomware-section
```

Each policy file must include complete frontmatter:

| Field | Required | Purpose |
|---|---|---|
| `title` | Yes | Policy name - shown on the site, in PDFs, and in reports |
| `description` | Yes | One-sentence summary shown in policy indexes |
| `extra.owner` | Yes | Person or team accountable for keeping the policy current |
| `extra.last_reviewed` | Yes | Date of last formal review (YYYY-MM-DD) |
| `extra.major_revisions` | Yes | At least one entry with date, description, author, approver, and version |

Policies under active drafting must set `draft: true` in their frontmatter. Draft policies are excluded from PDF generation (and hidden from the site index unless `show_draft_pdfs = true` is set in `config.toml`). The `draft` flag is removed when the policy is ready for approval.

The `{{/* org() */}}` shortcode must be used wherever the organization name appears in policy text. This ensures the name stays consistent and correct if it ever changes.

### 2. Review

When a draft is ready for review, the author opens a pull request (PR) against the main branch. The PR must include:

- A title summarizing the change
- A description explaining why the change is being made - the business or compliance reason, not just a description of what changed
- A link to any related incident, audit finding, or external requirement that prompted the change

Every PR requires at least one approval from a reviewer who did not author the change. The CISO (or designated delegate) must be a required reviewer for all policy files, enforced via a `.github/CODEOWNERS` file.

{{ org() }} acknowledges that not all approvers will be comfortable using the Git platform directly. Out-of-band review - sharing the PR diff over email or reviewing in a meeting - is acceptable, but the outcome must be recorded in the platform:

- The approver must formally approve the PR before it is merged
- Any discussion or decisions reached outside the PR must be summarized in a PR comment so the audit record is complete

The key principle is that the decision and its rationale must land in the PR. Verbal or email approvals alone are not sufficient.

### 3. Approval

A PR may be merged once all required approvals are in place and the automated build passes. Merging to main constitutes formal policy approval. The PR record - including the diff, reviewer identities, approval timestamps, and any comments - serves as the audit trail for the approval event.

Direct commits to the main branch are prohibited. Branch protection must be configured to enforce this.

### 4. Publication

The CI/CD pipeline runs automatically on every merge to main and:

1. Compiles all non-draft policies into PDFs
2. Builds the policy website
3. Deploys the site to the configured hosting platform
4. Uploads PDFs as build artifacts (retained for a minimum of 90 days)

For external distribution (auditors, customers, partners), a redacted build may be generated by triggering the pipeline with `redact_mode: true`. Redacted builds strip content enclosed in redact shortcode blocks. The decision to distribute a redacted build is made by the Security team on a per-release basis.

Draft watermarks may be added to any build by triggering the pipeline with `draft_mode: true`. PDFs stamped with a DRAFT watermark must not be distributed as final documents.

### 5. Periodic Review

All policies must be reviewed at least annually. The `extra.last_reviewed` field in each policy's frontmatter records when the last review occurred. A warning is displayed on the policy page when this date is more than one year in the past.

The Security team is responsible for auditing the policy library each quarter and initiating reviews for any policy approaching or past its annual review date. A review PR must be opened even if no substantive changes are made - the PR and updated `last_reviewed` date serve as the review record.

Triggered reviews - outside the annual cycle - are required when:

- Applicable laws, regulations, or contractual obligations change
- A security incident reveals a gap in existing policy
- A compliance audit identifies a control deficiency
- Significant changes to {{ org() }}'s technology environment or business processes make an existing policy inaccurate or incomplete

### 6. Version Control and Material Changes

The `extra.major_revisions` list in each policy's frontmatter is the canonical revision history. A new revision entry - and a version increment - is required when a policy change:

- Alters the scope of who or what the policy applies to
- Adds, removes, or modifies a compliance requirement or control
- Changes roles, responsibilities, or approval chains
- Would cause an auditor to ask "what changed and why?"

Version entries are **not** required for typo corrections, formatting changes, or periodic reviews where no substantive changes are made. Unnecessary version increments create audit noise and do not reflect actual policy changes.

Commit messages on policy changes must be descriptive enough for someone unfamiliar with the context to understand what changed and why. References to the triggering incident, audit finding, or control requirement should be included where applicable.

## Branch Protection Requirements

The main branch must be configured with the following protections:

- Pull requests required before merging - no direct pushes
- Minimum of two required approvals, including the CISO
- Stale approvals dismissed when new commits are pushed after review
- Status checks (CI build) must pass before merge
- These rules must apply to administrators as well as contributors

## Repository Access

Access to the policy repository is granted on a need-to-know basis and reviewed quarterly. The following access levels apply:

| Role | Access |
|---|---|
| Policy authors | Read + branch push (no merge to main) |
| Security team | Write (merge to main, manage branch protection) |
| CISO | Write; required reviewer via CODEOWNERS |
| All staff | Read access to the published site (access controls per deployment target) |

Deployment tokens and pipeline secrets must be rotated annually or immediately following a team access change.

## Sensitive Policy Content

Sections of policies that contain sensitive operational details - such as specific vendor names, internal tool configurations, key custodian contacts, or vulnerability-specific mitigations - may be wrapped in `redact` shortcode blocks. Content inside a redact block is visible on the website by default, but can be stripped from PDFs when the pipeline is triggered with `redact_mode: true`, and hidden from the website when `redact_web = true` is set in `config.toml`. This allows a single policy file to serve both internal and external distribution without maintaining separate documents.

The use of redaction tags is at the author's discretion, in consultation with the Security team. See the PolicyPress documentation for shortcode syntax.

## Roles and Responsibilities

| Role | Responsibility |
|---|---|
| Policy author | Draft new and revised policies; update frontmatter accurately |
| Policy owner (per policy) | Attest that policy content is accurate during annual review; approve major revisions |
| Security team | Review all PRs; maintain review schedule; manage repository access; merge approved changes |
| CISO | Approve all policy PRs (required reviewer); ratify the policy program annually |
| Senior leadership | Approve new policies and major revisions to foundational policies |

## Compliance and Enforcement

All {{ org() }} staff are expected to be familiar with and comply with policies relevant to their role. Publication of a policy to the policy site constitutes sufficient notice.

Violations of this policy - including making changes outside the PR workflow, merging without required approvals, distributing draft PDFs as final, or bypassing branch protection - may result in disciplinary action.

{% admonition(type="note") %}
The Git repository, PR history, and CI/CD build logs together constitute the audit trail for policy management. The main branch history must not be rewritten. Preserve repository access for at least three years to support audit requests.
{% end %}
